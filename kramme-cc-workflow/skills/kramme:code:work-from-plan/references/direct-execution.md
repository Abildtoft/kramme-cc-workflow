# Direct Execution Reference

Direct execution is for bounded current-branch work only. It borrows the planning and verification discipline of issue implementation without creating a competing tracker.

## Direct Workflow

1. **Context check**
   - Run `git status --porcelain` and `git branch --show-current`.
   - Identify whether existing uncommitted changes overlap the plan.
   - Read applicable project instruction files before choosing verification commands.

2. **Codebase exploration**
   - Search for files named in the plan.
   - Search for similar implementations with `rg` and `rg --files`.
   - Read nearby tests, fixtures, documentation, scripts, or config that govern the affected area.
   - Identify the smallest set of files that satisfy the plan.

3. **Implementation plan**
   - Create a short todo list from the plan's completion criteria.
   - Order tasks so each step leaves the codebase understandable and preferably verifiable.
   - State the simplest implementation that could satisfy the plan.

4. **Implement**
   - Edit only files needed for the current task.
   - Preserve existing local patterns and terminology.
   - Add focused tests when behavior changes or when the risk justifies coverage.
   - Do not add branch, PR, CI, release, or issue-tracker automation.

5. **Verify**
   - Run the narrowest useful checks after meaningful changes.
   - Run final verification using `kramme:verify:run` when available.
   - If that skill is unavailable, run the discovered check-only project commands directly.
   - Continue through all relevant checks and report skipped checks with reasons.

6. **Close out**
   - Summarize changed files and behavior.
   - Report verification commands and results.
   - List non-goals left untouched.
   - Surface remaining risk honestly.

## Missing Requirement Rules

Use `MISSING REQUIREMENT` instead of guessing when a missing fact affects implementation safety.

Examples:

```
MISSING REQUIREMENT: the plan references a Linear issue but does not include the issue identifier.
MISSING REQUIREMENT: completion criteria are absent for a user-facing behavior change.
MISSING REQUIREMENT: no verification command is discoverable for production code changes.
MISSING REQUIREMENT: prerequisite W01A is required and no equivalent local guidance exists.
```

After emitting `MISSING REQUIREMENT`, ask the smallest follow-up question or name the best existing workflow to use next.

## Direct Work Stop Conditions

Stop direct execution and re-route when:

- The work becomes multi-phase.
- The implementation needs an architecture or product decision not present in the plan.
- The plan's non-goals conflict with the necessary fix.
- Affected files are missing or generated by a tool that should be run instead.
- Verification fails for reasons outside the plan's scope and fixing them would be unrelated.

Use `NOTICED BUT NOT TOUCHING` for unrelated observations:

```
NOTICED BUT NOT TOUCHING: adjacent docs row is stale.
Why skipping: outside this plan's completion criteria.
```
